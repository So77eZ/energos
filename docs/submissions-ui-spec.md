# Заявки на добавление энергетика — UI/форм-спека для дизайнера

Что должно быть на формах фичи «заявки на добавление». Только про UI: поля, контролы, состояния, тексты. Бэкенд-контракт и дев-задачи — в [improvements.md](../improvements.md).

Фича = 4 поверхности. Заявка живёт по статусам: `pending → approved` / `rejected`. После approve бэк создаёт **черновик** напитка.

---

## A. Форма заявки (user) — `/submit`

**Цель:** юзер предлагает энергетик, которого нет в каталоге. Минимум трения — большинство полей опциональны.

**Поля:**

| Поле | Тип | Обяз | Правила | Плейсхолдер / подсказка |
|---|---|---|---|---|
| Название | text | **да** | 2–80 символов, счётчик | «Например: BURN Mango Loco» |
| Без сахара | toggle / switch | **да** | дефолт — выкл (с сахаром) | «Без сахара / zero-sugar» |
| Цена | number | нет | > 0, шаг 0.01, ₽ | «89.00» · «средняя по магазинам» |
| Комментарий | textarea | нет | ≤ 500, счётчик | «Где видел, ноты вкуса, чем интересен» |
| Фото банки | upload | нет | jpg/png, ≤ 5 МБ, drag-n-drop + клик | «Перетащи или кликни» |

**Контролы:** кнопка «Отправить заявку» (disabled пока невалидно), «Отменить». У фото — превью + «убрать».

**Состояния:**
- Невалидно → submit disabled, инлайн-ошибки под полем («минимум 2 символа», «введи число > 0», «файл больше 5 МБ»).
- Отправка → лоадер на кнопке.
- Успех → тост «Заявка №N отправлена — ждёт модерации», увод/намёк в **Профиль → Мои заявки**.
- Ошибка / rate-limit → тост с текстом ошибки.
- Не авторизован → форма недоступна, CTA «Войти, чтобы предложить».

**Бейдж-нотис внизу:** «Заявка пройдёт модерацию. Статус — в Профиль → Мои заявки».

---

## B. Мои заявки (user) — `/profile`

**Цель:** read-only трекинг статуса своих заявок. Без действий (или только «отменить» pending, если бэк добавит DELETE).

**Карточка / строка:** название, дата отправки, бейдж статуса (в очереди / одобрено / отклонено), для rejected — **причина отклонения** текстом.

**Состояния:** пусто → «Ты ещё не предлагал напитки» + CTA на `/submit`.

---

## C. Очередь модерации (admin) — `/admin` → таб «Заявки»

**Цель:** разбор очереди. Уже есть скелет.

**Шапка:** фильтр-табы со счётчиками — **В очереди · Одобренные · Отклонённые · Все**. Дефолт = «В очереди».

**Дашборд-карточка** «Заявки в очереди» (N из M) — счётчик pending.

**Состояния:** пустой фильтр → empty-state (колба + «Нет заявок в этом разделе»).

---

## D. Карточка заявки + действия (admin)

**Карточка `pending`:** фото банки (или плейсхолдер «нет фото»), название, автор (аватар + ник), дата, цена, комментарий юзера, кнопки **Отклонить / Одобрить**.

**Карточка терминальная** (approved / rejected): без фото (бэк чистит картинку после решения), бейдж статуса, дата решения, для rejected — причина. Без кнопок.

### D1. Модалка «Одобрить»

- Заголовок «Одобрить заявку?»
- Текст: «Будет создан **черновик** напитка «{название}». После одобрения откроется карточка — дозаполни вкусовые метрики и сохрани.»
- Кнопки: «Одобрить» (primary) / «Отмена».
- **После подтверждения → редирект на форму редактирования черновика** (см. E).

### D2. Модалка «Отклонить» — новая, сейчас причина захардкожена

- Заголовок «Отклонить заявку?»
- **Поле «Причина» (textarea, ≤ 500)** — видна автору в «Мои заявки». Сейчас прибито «Не прошло модерацию» — нужно живое поле.
- Опц. чипы-пресеты для скорости: «Дубль», «Не энергетик», «Мало данных», «Спам/мусор» → кликом подставляют текст.
- Required? — рекомендую обязательной (юзеру нужен фидбек). Дизайнеру решить.
- Кнопки: «Отклонить» (danger) / «Отмена».

---

## E. Черновик после approve (admin) — `/admin/drinks/{id}/edit`

После approve бэк создаёт черновик `EnergyDrink`. Админ попадает в **существующую форму редактирования напитка**, часть полей предзаполнена из заявки.

**Предзаполнено из заявки:** название, цена, фото, без сахара.

**Дозаполнить (пусто, обязательно перед публикацией):**
- 6 вкусовых метрик (кислость, сладость, интенсивность, газировка, послевкусие, ценность) — шкала 1–5.
- Бренд, объём, описание (если эти поля будут — см. backlog improvements.md).

**Состояние черновика:** дизайнеру решить, нужен ли визуальный маркер «черновик / не опубликован» в каталоге и в админ-списке напитков, пока метрики не заполнены.

> Уточнить у бэка: попадает ли черновик сразу в публичный каталог (с нулевыми метриками) или есть флаг `is_draft` / `published`, скрывающий его до дозаполнения. Это меняет дизайн состояния «черновик».
