## Актуальные задачи

### 🔴 Высокий приоритет

- **Изображения не отображаются на мобильных устройствах** — при открытии сайта с телефона картинки энергетиков не загружаются.

### 🟡 Средний приоритет

- **Сортировка по цене** — в фильтре не хватает сортировки по дороговизне.
- **Сортировка и фильтрация отзывов** по рейтингу и дате. На странице напитка под отзывами `<select>` со стилем `.select-min` (Сначала новые / Сначала старые / Высокая оценка / Низкая оценка) — частично уже есть, не хватает фильтрации.
- **Полнотекстовый поиск** — нечёткий поиск через `pg_trgm` (монстер / монстэр / monster).
- **Поиск независимо от порядка слов** — сейчас `name.toLowerCase().includes(q)` ищет подстроку как есть: запрос «burn киви яблоко» не найдёт напиток «BURN Яблоко, киви». Нужно разбивать запрос на токены и проверять что КАЖДЫЙ токен встречается в имени (в любом порядке). Решается на клиенте splitом `q.split(/\s+/)` и `tokens.every(t => name.includes(t))`, либо на бэке через `to_tsvector` / `pg_trgm` если будет полнотекстовый поиск (см. соседний пункт — они комплементарны).
- **Добавить способ обратной связи** — ссылка на тг-бота или личку.
- **Продуктовая аналитика — какие разделы используются, какие игнорируют** — Яндекс Метрика уже подключена для общей посещаемости (см. archive), но нужны структурированные events для понимания UX. Замечание от бэкендера: «каким разделом пользуются, а каким нет». **Стек:** через `ym(id, 'reachGoal', 'event_name')` или PostHog/Plausible если хочется отдельный dashboard. **Что трекать минимум:**
  - переходы между основными разделами (catalog / drink / compare / tier / glossary / submit / profile / admin)
  - использование view-switch (`grid` vs `heat`) и фильтр-popover (какие тиры/цены/toggle'ы реально кликают)
  - открытие TweaksPanel и выбор темы/акцента/шрифта (популярные комбинации)
  - submit-форма: процент дроп-офф'ов на каждом поле, успех/отказ
  - favorites/emoji toggle — engagement signals
  - search queries (без PII — только сами строки) — что ищут, что не находят
  
  **Реализация:** небольшая обёртка `track(event, props?)` в `shared/lib/analytics.ts`, дёргается из ключевых обработчиков. Респекта `prefers-reduced-motion` тут нет, но cookie-consent на ЕС-юзеров — да (выключать если не согласились).
- **Карта вкусов: выбор осей X/Y** — сейчас отображаются только 2 метрики из 6.
- **`loading.tsx` / Suspense-границы** — Next.js App Router поддерживает стриминг: страницы могут показывать скелетон пока данные грузятся, вместо белого экрана. Сейчас ни одного `loading.tsx` нет.
- **`not-found.tsx`** — если передать несуществующий `?id=` в адресе, страница молча показывает первый напиток. Нужна явная 404-страница для отсутствующих напитков.
- **Бренд напитка как отдельное поле в БД** — сейчас фронт берёт первый ALL-CAPS токен из имени (`RED BULL Red Edition` → бренд `RED BULL`), fallback `ENERGOS`. С нормальным полем `brand` всё было бы стабильнее и можно фильтровать.
- **Объём напитка как отдельное поле** — `0.45 л`, `0.25 л` сидит в имени. `cleanDrinkName()` его выкидывает, но потерянная информация полезная — фильтр по объёму и группировка дублей по объёму невозможны.
- **Описание/blurb напитка** — поле для hero/карточки, 1–2 предложения о вкусе/составе. Сейчас место в дизайне есть, но в БД пусто.
- **Аватарки пользователей** — сейчас аватар везде это цветной кружок с первой буквой ника (`pickAvatarColor` от хеша user_id, буква = `username[0]`). Пора дать возможность загружать настоящие.

  - `DELETE /api/auth/me/avatar` (auth, 204) — снять (вернуться к буквенному).
  - Хранилище — то же что и для drink-изображений (видимо `/uploads/avatars/{user_id}.{ext}` с очисткой старого файла при замене).
  - В `UserResponse` добавить `avatar_url: str | None`.

  **Фронт:**
  - В `entities/user/api/authApi.ts` добавить `uploadAvatar(file, token)` и `removeAvatar(token)`.
  - Универсальный компонент `<Avatar user={user} size={40} />` который проверяет `user.avatar_url`: если есть — `<img>`, иначе — цветной кружок с буквой (текущая логика). Заменить инлайн-кружки в:
    - [HeaderAvatar.tsx](frontend/src/widgets/header/ui/HeaderAvatar.tsx)
    - `prof-avatar` в [ProfilePage.tsx](frontend/src/widgets/profile/ui/ProfilePage.tsx)

    - `lb-avatar` в [LeadersTab.tsx](frontend/src/widgets/admin-page/ui/tabs/LeadersTab.tsx)
  - В профиле — клик по аватару открывает file picker; preview; кнопка «Сохранить» / «Удалить аватар» / «Отмена». После upload — обновить `useCurrentUser()` контекст и `router.refresh()`.
  - Опционально: crop-step через `react-easy-crop` (~30KB) — обрезать в квадрат 256×256 перед отправкой. Иначе бэк сам круглит через CSS `border-radius: 50%`.

  **Опционально (на потом):** Gravatar-fallback по email (если будем хранить email — см. пункт «Сброс пароля / верификация email»), или DiceBear identicons для неавторизованных/новых юзеров — детерминистично от user_id, чтобы у каждого был свой узнаваемый рисунок ещё до того как загрузил настоящую.
- **Сохранение страницы пагинации при back-навигации** — листаешь каталог, открываешь напиток на 4-й странице, жмёшь «Назад» — снова на 1-й странице. Сейчас в `DrinkCatalog` `const [page, setPage] = useState(1)`: локальный React-state теряется при unmount, browser back восстанавливает HTML, но page = 1.

  **Фикс:** перенести `page` в URL через `useSearchParams`:
  - Читать `Number(searchParams.get('page')) || 1` для начального значения
  - При смене страницы — `router.replace('/?page=' + p, { scroll: false })`
  - Уже есть готовый паттерн в `widgets/compare-page/ui/ComparePage.tsx` (там state ↔ URL для a/b/c слотов)
  - Бонус: ссылки на конкретные страницы каталога становятся шарящимися (`/?page=4`)
  - Ещё бонус: можно поднять hero только на page=1 (на остальных страницах он засоряет)

  **Заодно стоит** перенести в URL `sort`, `noSugarOnly`, `search`, `tiers`, `priceRange`, `onlyNew`, `view` (всё что в `useCatalogSearch`) — тот же баг с потерей фильтров уже отмечен в тех-долге.

### 🟢 Низкий приоритет / По желанию

- **3D-банки по бокам каталога — улучшения** — банки уже работают (видны при `min-width: 1440px`), нужны доработки:
  - **Brекпоинт-чувствительность**: на мобиле полностью скрывать (`display: none !important`), чтобы не пожирать батарею
  - **Цвет акцентной полоски привязать к `--accent`** из TweaksPanel — сейчас захардкожено
  - **Улучшение после**: заменить абстрактные банки на 3D топ-2 напитка по среднему рейтингу (брать с бэкенда — нужно `can_colors` поле или генератор от хеша id)

- **Геймификация 3D-банок: «разгон до взрыва» по кликам** — сейчас банки крутятся быстрее при наведении/удержании (state-machine `idle → spinUp → maxed → burst → hidden → fadeIn` в `ThreeCans.tsx`). Заменить трекинг hover на трекинг кликов: каждый клик добавляет к **угловому ускорению** (не скорости), скорость растёт интегралом — то есть один клик чуть ускоряет, серия кликов раскручивает к взрыву экспоненциально. Без кликов ускорение затухает в 0 (демпфер), банка плавно возвращается к idle. Завязать на achievement-систему (см. соседний пункт про ачивки): «Взорвать N банок» / «Раскрутить за <2с» / «Каскад из 5 взрывов подряд».

- **Сканирование штрих-кода банки → переход на карточку** — пользователь подносит банку к камере телефона, приложение распознаёт EAN-13, ищет напиток в нашей базе и редиректит на `/drinks?id=N`. Если не нашли — предлагает оставить заявку на добавление (см. соседний пункт «Заявки на добавление энергетиков»).

  **Бэк:**
  - Миграция: новая колонка `barcode: str | None` в `energy_drinks`, уникальный partial-index `WHERE barcode IS NOT NULL` (один EAN — одна позиция; nullable потому что для большинства старых импортов будет пусто пока не добьём вручную).
  - `GET /api/energy-drinks/by-barcode/{ean}` — 200 `EnergyDrinkSchema` или 404. EAN валидируется регуляркой `^\d{8,14}$` (EAN-8/EAN-13/UPC-A/ITF-14).
  - В `CreateEnergyDrinkSchema` / `UpdateEnergyDrinkSchema` добавить опциональное поле `barcode`. Для админки.
  - При импорте из DNS подтянуть `barcode` если он есть в источнике (DNS отдаёт его на карточке товара).

  **Фронт:**
  - Кнопка «📷 Сканировать» в шапке (рядом с поиском) или floating action на мобиле (камера полезнее в стоячем магазине, чем за десктопом).
  - Библиотека: `@zxing/library` (~50 КБ gz, ES-модули, открывает камеру через `navigator.mediaDevices.getUserMedia`, читает EAN-13/UPC/Code128). Альтернатива — `html5-qrcode` с готовым UI, но добавляет ~40 КБ за приятный intro-фрейм. Я бы взял zxing напрямую и сверстал свой viewfinder.
  - HTTPS обязательно — без него Camera API заблокирован браузером. У нас Caddy в проде это даёт; локально работает только на `localhost` или через ngrok.
  - Флоу:
    1. Клик «Сканировать» → modal-overlay с `<video>` (BackCamera через `facingMode: 'environment'`).
    2. Полупрозрачная рамка-ROI в центре + подсказка «Наведите штрих-код в рамку».
    3. zxing-loop на каждом кадре → при первом успехе: бип (`new Audio('...')`) + лёгкая вибрация (`navigator.vibrate(50)`).
    4. Запрос на `GET /api/energy-drinks/by-barcode/{ean}` → success → `router.push('/drinks?id=' + id)`.
    5. 404 → показать «Этот напиток не в каталоге» + CTA «Предложить добавить» (откроет форму заявки с уже подставленным EAN).
  - Fallback для desktop / без камеры / при отказе в permission: ручной input «Ввести код вручную».
  - Edge cases: одна банка разных объёмов (0.25/0.5 л) — у каждой свой EAN, в БД будут разные `id`. Это ОК — каждый EAN ведёт на свою позицию.
  - Опционально (улучшение): после успешного скана засчитывать «реальное взаимодействие» в счётчик активности юзера → влияет на ачивки.

Современный стандарт безопасности (нет утечек хешей, нет проблемы reused passwords) и удобства (не надо помнить/восстанавливать). Флоу: «Введи телефон/email → получи 6-значный код → ввёл → залогинен».

  - Старый password-based вход можно оставить как fallback на переходный период (или сразу выпилить — миграция всех существующих юзеров через email-link).

#### Геймификация и онбординг

Тематический блок — улучшения вовлечённости. Большинство пунктов работают независимо, можно выкатывать постепенно.

- **Интерактивный тур по сайту для новых пользователей** — после age-gate'а на главной появляется ненавязчивый prompt в углу: «Пройти короткий тур по сайту?» с кнопками «Окей» / «Пропустить» / `[ ] Не показывать снова`. Если «Окей» — последовательность подсветок с tooltip'ами.

  **UX:**
  - Spotlight: затемнить весь экран overlay'ем (`rgba(0,0,0,.7)`), кроме подсвечиваемого элемента — кольцо вокруг цели, остальное dim'нуто. `pointer-events: none` на overlay'е, но клики через него до подсвеченного элемента не доходят (backdrop ловит и игнорирует) — иначе юзер случайно прокликнет настоящие кнопки.
  - Tooltip auto-positioning: top/bottom/left/right в зависимости от свободного места и размера экрана. Arrow указывает на цель.
  - На каждом шаге: текст подсказки + кнопки «← Назад», «Окей →», «Пропустить тур». Индикатор прогресса «3 / 8».
  - Шаги тура: каталог-сетка → SortBar → view-switch grid/heat → фильтр-popover → «Не нашёл напиток?» CTA → side-rail с tier-list → tweaks-fab → favorites-кнопка на карточке (только для авторизованных). При смене страницы (например на /compare) тур запоминает шаг и продолжает.
  - Сегментация: разные туры для гостя / залогиненного юзера / админа.

  **Триггер запуска:**
  - При первом заходе после age-gate, если `localStorage.energos_onboarding_done` не выставлен.
  - Reset из TweaksPanel или из профиля кнопкой «Запустить тур заново» — полезно при апдейтах.

  **Стек:** можно нативно (`<dialog>` + position-fixed overlay + custom hook), ~150 строк. Или библиотека: `driver.js` (~10KB), `intro.js`, `react-joyride`.

- **XP + уровни** — действия дают XP: отзыв = 10, отзыв с комментарием = 15, заявка одобрена = 25, добавил в избранное = 1, получил эмодзи-реакцию на свой отзыв = 2. Уровень растёт квадратично (`level = floor(sqrt(xp / 50))`). XP-полоса в hero профиля.

  **Бэк:** новые поля `users.xp INT DEFAULT 0`, либо денормализация через триггеры/views. Endpoint в `/api/auth/me/` отдаёт `xp` и `level`.

- **Бейджи и звания (эскалация)** — пятиступенчатая система с конкретными условиями:
  1. **Новичок** — 1 отзыв
  2. **Дегустатор** — 5 отзывов
  3. **Критик** — 5 отзывов с комментариями
  4. **Знаток** — 10 отзывов с комментариями
  5. **Активист** — 5 эмодзи-реакций поставленных другим
  6. **Эксперт-сладкоежка** — преобладание сладких оценок (avg sweetness > 4)
  7. **Полночная сова** — 5+ отзывов между 0:00 и 4:00 по локальному времени
  8. **Первопроходец** — оставил отзыв на напиток, у которого не было других отзывов
  9. **Универсал** — отзывы хотя бы на один напиток каждого тира S/A/B/C/D
  10. **Топ-10%** — попадание в верхние 10% по числу отзывов

  Бейджи отображаются на профиле и (топ-3) рядом с никнеймом в карточке отзыва. Прогресс к следующему виден в табе «Достижения» (текущая `AchievementsTab` уже подходит для этой UI-формы — расширить с 8 моков до настоящих).

- **Серия дней с отзывами (streak)** — `MAX(streak)` подряд идущих дней с хотя бы одним отзывом. Прерывается при пропуске суток. На профиле — огонёк с числом «🔥 12» рядом с никнеймом, тултип «Не прерывай серию!». Если серия 0 — иконка серого цвета.

  **Архитектурное замечание от бэкендера:** не считать в рантайме (full-scan таблицы отзывов на каждом запросе профиля = боль на реальных объёмах данных, даже с индексом). Правильно — **отдельная cron-job'а** ночью пересчитывает streak для всех юзеров с активностью за последние N дней и кладёт в денормализованное поле `users.review_streak: int`. Endpoint `/me/` просто читает это поле. При создании отзыва инкрементить можно прямо в моменте (если предыдущий отзыв был вчера → +1, иначе → 1) — это O(1).

- **Меню квестов (Daily challenge + Quest tracker)** ★ — единая страница `/quests` (или вкладка в профиле) с двумя секциями. **Отмечено бэкендером как наиболее интересная из геймификационных фич.**
  - **Сегодняшнее задание** — псевдослучайное (детерминистично от `date + user.id`): «Оставь отзыв на напиток из тира C», «Сравни 2 напитка с одинаковым brand», «Добавь 3 напитка в избранное». Респаун в полночь. За выполнение +25 XP.
  - **Длинные квесты** — многошаговые цепочки с прогресс-баром: «Открой 5 разных напитков → выбери любимый → оставь отзыв → добавь в избранное». Состояние хранится на бэке, отслеживается через события (или прямые запросы агрегатов).

- **Calendar heatmap активности в профиле** — GitHub-style сетка дней за последние 12 месяцев, цвет ячейки пропорционален числу отзывов в этот день (0 → серый, 4+ → акцентный цвет). Tooltip при наведении: «2026-03-12 · 3 отзыва». Считается из `created_at` отзывов юзера.

- **Совпадение вкусов между пользователями** — для каждой пары юзеров считается L2-расстояние по средним метрикам в их отзывах. На профиле другого юзера (если когда-то будет публичный профиль) показывается «У вас 87% совпадение». Также можно использовать для «Похожие на тебя рецензенты» на твоём профиле.

- **Tasting blind game** — мини-игра доступная с главной или из профиля. Показывается только hex-радар метрик случайного напитка (без названия и канки), и 4 кнопки с именами напитков (1 правильный + 3 рандомных). Угадал — +5 XP, серия из 5 правильных подряд = «Эксперт-сомелье» бейдж. Очки и история ответов хранятся в `quiz_attempts` (или просто localStorage для MVP).

- **Random drink / Gachapon** — кнопка «🎰 Случайный напиток» (в шапке или на главной). При клике — анимация крутящегося колеса с превью разных напитков, замедляется и останавливается на случайном. Редирект на его карточку. Простая визуальная радость, без бэк-логики.

- **Easter eggs** — три штуки для атмосферы:
  - **Konami code** (↑↑↓↓←→←→BA) → секретный «retro»-режим: меняется тема на CRT-зелёный, включается максимум scanlines и grain, шрифт принудительно Monocraft. Toast «🕹 SECRET MODE». Сохраняется в session, сбрасывается при перезагрузке.
  - **100 кликов по логотипу ENERGOS** → юбилейная анимация: банка-фейерверк в центре экрана, тост «Ты кликнул логотип 100 раз. У тебя слишком много свободного времени», скрытый бейдж «Логотипоман».
  - **10 спрятанных молний** ⚡ разбросаны по сайту (на странице drink, glossary, /tier, hero и т.д.) — мини-иконки разного цвета, не относящиеся к основному UI. Клик коллекционирует. Сбор всех 10 даёт бейдж «Энергетик-следопыт».

- **«Топ X% рецензентов» — упрощённая версия «влиятельного рецензента»** — без сложного веса голосов. Просто метка на профиле: если ты в верхних 10% по числу отзывов — отображается «🥇 Топ-10%», 25% → «🥈 Топ-25%», 50% → «🥉 Топ-50%». Считается на бэке percentile по `review_count DESC` через `NTILE(10) OVER (ORDER BY review_count DESC)`. Дёшево, дает социальный капитал.

- **Социальный граф (на потом)** — подписки на других юзеров, лента отзывов от подписок, уведомления когда любимый напиток получил новый отзыв. Большая фича — нужны таблицы `user_follows`, `user_notifications`, отдельные endpoints, и страница `/feed`. Целесообразно когда базовая геймификация (XP/бейджи/streak) уже работает и есть активная аудитория.

#### Из миграции дизайна (пропущено или замокано)

Дизайн содержал ряд элементов, для которых нет данных на бэке либо это новая фича — оставлены на потом:

- **Live ticker в шапке** — лента «X оценил Y · 4.5★ · 2m назад» в подвале хедера. Дизайн готов (`.ticker`/`.ticker-row`/`.ticker-item`), но нужен endpoint `/api/recent-events` или WebSocket — взять последние ~20 действий (созданные/обновлённые отзывы) с usernames и drink names.
- **Email-поле и «Войти как гость (demo)»** — в auth-форме дизайна были `<input type="email">` и кнопка demo-входа. Пропущены — наш бэк аутентифицируется по `username + password`, demo-режима тоже нет. Email понадобится для верификации/сброса пароля (см. отдельный пункт).
- **«AI: похожие по вкусовому профилю»** — на drink-странице блок назван AI, но фактически считает евклидово расстояние L2 по 6 метрикам. Если будет нужен настоящий контент-бейзед/коллаборативный — потребуется отдельный сервис.
- **«Популярное сейчас» вместо «AI: подобные вкусы» в SideRail** — в дизайне это была персонализированная рекомендация под пользователя («для marina_x» — топ-3 близких по профилю). У нас вместо неё топ-3 по числу отзывов — без персонализации.
- **Tier как сохраняемое поле в БД** — сейчас `tierFromRating()` вычисляет S/A/B/C/D из рейтинга. С админ-полем можно курировать вручную, не привязываясь к рейтингу (для tier-list страницы).
- **CTA «Сравнить» в hero напитка** — был в дизайне, но без `/compare` страницы не имеет смысла. Появится вместе с фичей сравнения напитков (`/compare` уже есть, кнопку добавить).
- **`joined`-дата пользователя на профиле** — в макете «с дд.мм.гггг», нет поля `created_at` на `User`. Нужно добавить миграцию.
- **`drink.can` colors (цвета 3D-банки)** — на каждый напиток в макете была уникальная цветовая раскраска банки (body/stripe/accent). Сейчас цвет вычисляется из blend-цвета метрик, остальное — мок. Если хотим уникальные банки — нужно `can_colors` поле или генератор от хеша id.

### 🔧 Технический долг

- **Баг: `sort` и `noSugarOnly` не сбрасываются при смене маршрута** — `SearchResetter` в `catalog-search.tsx` сбрасывает `search`, `searchItems`, `filterOpen`, но забывает про `sort` и `noSugarOnly` (и новые `tiers`/`priceRange`/`onlyNew`/`view`). Если выбрать «Сначала дороже» и перейти на другую страницу — фильтр остаётся висеть.
- **`avgReview` использует `userReviews[0]` как базу** — в `ReviewsPage` средний отзыв считается через `{ ...userReviews[0], ...metrics }`, из-за чего в объект попадают `id`, `user_id`, `username` первого пользователя. Лучше собирать объект явно.
- **`deleteReviewAction` молча игнорирует ошибки** — если удаление не прошло (сеть, 403), пользователь не получит никакого сигнала, страница просто обновится как ни в чём не бывало.
- **Все Google Fonts грузятся всегда** — в `layout.tsx` прописаны все шрифты сразу (~200 KB лишних запросов), независимо от того какой выбрал пользователь. Можно грузить только активный.
- **`FONT_INIT_SCRIPT` не валидирует значение шрифта** — inline-скрипт применяет `p.font` как есть, без проверки что это один из допустимых вариантов. Если в localStorage окажется мусор — он будет применён к `--font-sans`.
- **`enrichDrinks` дважды на drink-странице** — `DrinkPage` вызывает `enrichDrinks(drinks, allReviews)` для SimilarRail, а `DrinkCatalog` ещё раз делает то же на главной. Если страницы соседствуют — можно вынести в провайдер/кэш.
- **Compatibility-shim `.glass` / `.glass-surface`** — оставлены в `globals.css` для оставшихся legacy-консьюмеров (admin/taste-map/auth и т.д.). Удалить после миграции остальных страниц.
- **N+1 запросов `/emojis/` под каждым отзывом** — `EmojiBar` фетчит реакции на mount, по одному запросу на каждый отзыв. На странице с 20 отзывами это 20 параллельных GET'ов к `/api/reviews/{id}/emojis/`. **Фикс на бэке:** добавить агрегат `emoji_summary: [{ emoji, count, user_ids }]` (или `[{ emoji, count, mine }]` если знаем юзера через token) в основную схему `EnergyDrinkReviewSchema`. После этого `EmojiBar` принимает `initialReactions` пропом и не дёргает API на mount.
- **Emoji-picker позиционируется только вверх (`bottom: calc(100% + 6px)`)** — если отзыв находится близко к верху страницы (например первый видимый), picker может уйти за viewport. Flip-logic (выбор top/bottom в зависимости от свободного места) не реализован.
- **Мусор `energos_favorites_mock` в localStorage** — старый mock FavoritesProvider писал избранное в `localStorage[energos_favorites_mock]`. После миграции на API-based провайдер этот ключ больше не читается, но у тех кто пользовался mock — остался в браузере. Можно добавить one-shot cleanup в FavoritesProvider на mount: `localStorage.removeItem('energos_favorites_mock')`.
- **Monocraft.ttc — 5.9MB** — Monocraft подключён через `@font-face` с `.ttc` (TrueType Collection). Большой файл — конвертнуть в `.woff2` сэкономит ~75% (1.5MB вместо 5.9). Команда: распаковать `Monocraft-ttf.zip` из релиза → `woff2_compress Monocraft.ttf` (или онлайн-конвертер). Заменить src в globals.css на `url('/fonts/Monocraft.woff2') format('woff2')` и удалить .ttc.
- **Анимация перехода dark ↔ light** — сейчас при переключении темы flash. Можно `transition: background-color .3s, color .3s` на root, либо View Transitions API.
- **`prefers-reduced-motion`** — полностью отключать для юзеров с этой preferences: liquid-bg, scanlines, 3D-cans hover-animations. Оставить только базовые fade/scale переходы.
- **`base64` фото в payload заявки — может вылететь 1+ МБ** — сейчас фото в `/api/submissions` кодируется base64 и идёт в JSON. На многомегабайтных снимках это излишне раздувает payload. Альтернатива: загрузка отдельным шагом `POST /uploads` → получает `photo_url` → `POST /submissions` с этим url. Либо multipart-форма.

### 💬 Обсуждения

- **Убрать из админки оставление отзывов** — администратор должен оставлять отзывы через обычный интерфейс, а не отдельную форму в панели управления.
